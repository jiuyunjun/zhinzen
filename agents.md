# AGENTS.md

## 1. 项目目标

本项目是一个位置共享 Web / App 产品，用于解决两人及以上场景下，互相实时知道对方位置、方向、距离和移动轨迹的需求。

核心目标：

1. 手机浏览器优先支持。
2. 后续提供 Android App。
3. 再后续移植 iOS App。
4. Web 和 App 保持一致的 UI 风格。
5. 后端优先使用 GCP / Firebase。
6. 不设计账号体系。
7. 用户只需要在自己的客户端输入姓名。
8. 当前设备直接作为一个用户参与位置共享。
9. 每次代码或文档修改完成后必须 commit。
10. 所有文件读写必须严格使用 UTF-8。

## 2. Agent 分工

### 2.1 UI Design Agent

使用：

* Claude Design

职责：

1. 负责 Web 和 App 的 UI / UX 设计。
2. 优先设计手机浏览器布局。
3. 保证 Web、Android、未来 iOS 的视觉风格一致。
4. 输出可实现的设计稿、组件结构、页面状态和交互说明。
5. 所有设计必须考虑真实手机屏幕上的单手操作体验。
6. 所有地图相关 UI 必须考虑地图上覆盖控件的可读性。
7. 所有方向、距离、在线状态必须有清晰的视觉表达。

UI 设计要求：

1. 移动端优先。
2. 地图为主界面核心。
3. 地图之外也必须有成员列表或成员图标区。
4. 当前用户、在线用户、离线用户、正在移动用户需要有不同状态表达。
5. 点击用户图标后，需要进入对方详情面板。
6. 对方详情面板需要支持：

   * 查看对方当前位置
   * 查看对方移动轨迹
   * 跳转 Google Maps 导航
   * 显示方向指针
   * 显示距离
   * App 侧显示 UWB / 蓝牙状态
7. 设计应尽量避免复杂设置。
8. 用户不需要理解账号、登录、注册等概念。

## 3. 文件读写规则

所有 Agent 必须遵守：

1. 所有文本文件统一使用 UTF-8。
2. 读取文件时明确使用 UTF-8。
3. 写入文件时明确使用 UTF-8。
4. 不允许使用系统默认编码。
5. 不允许生成乱码。
6. 不允许混用 Shift-JIS、GBK、UTF-16 等编码。
7. 修改中文、日文、英文混合文件时，必须保持 UTF-8。
8. 生成 Markdown、JSON、YAML、TypeScript、Kotlin、Swift、HTML、CSS 文件时，必须保持 UTF-8。

推荐实现方式：

* Node.js：显式使用 `utf8`
* Python：显式使用 `encoding="utf-8"`
* Kotlin：显式使用 `Charsets.UTF_8`
* Swift：显式使用 `.utf8`

## 4. Git 提交规则

### 4.1 基本规则

每次修改完成后必须 commit。

禁止：

1. 修改完成后不 commit。
2. 多个无关需求混在一个 commit。
3. commit message 写得过于模糊。
4. 未测试就声称完成。
5. 修改文档后忘记提交。
6. 修改代码后忘记提交。
7. 自动格式化大量无关文件。

---

### 4.2 commit message 格式

优先使用 Conventional Commits。

格式：

```text
<type>(<scope>): <summary>
```

常用类型：

```text
feat: 新功能
fix: 修复问题
docs: 文档修改
style: 样式修改，不影响逻辑
refactor: 重构
test: 测试相关
chore: 构建、配置、工具相关
```

示例：

```text
docs(design): 补充设备即用户模型
feat(web): 添加地图首页布局
feat(web): 添加设备名称输入流程
feat(firebase): 添加房间和成员数据结构
fix(web): 修复位置权限拒绝时的提示
feat(android): 添加 UWB 能力检测
```

## 5. 项目技术方向

### 5.1 后端优先级

优先使用：

1. Firebase
2. GCP
3. Google Maps Platform

默认后端方案：

1. Firebase Hosting：部署 Web。
2. Firebase Firestore：保存房间、成员、轨迹摘要、共享状态。
3. Firebase Realtime Database：处理在线状态和实时位置。
4. Firebase Cloud Functions：处理房间清理、过期数据删除、服务端校验。
5. Firebase App Check：降低非法客户端调用风险。
6. Google Maps Platform：地图显示、地点显示、导航跳转。
7. Cloud Storage：后续可存储日志、导出数据或调试文件。

---

### 5.2 不使用账号体系

本项目不设计用户账号。

用户体验上：

1. 用户打开 Web 或 App。
2. 输入自己的显示姓名。
3. 客户端生成本设备的 `deviceId`。
4. 客户端把当前设备作为一个参与者加入房间。
5. 用户通过邀请链接或房间码与其他人共享位置。
6. 不要求注册。
7. 不要求登录。
8. 不要求邮箱。
9. 不要求手机号。
10. 不要求密码。

技术实现上：

1. `deviceId` 在客户端本地生成。
2. `deviceId` 保存在本地存储中。
3. 同一设备再次打开时复用同一个 `deviceId`。
4. 用户可以修改显示姓名。
5. 后端根据 `roomId + deviceId` 区分参与者。
6. 为了防止别人伪造同一个 `deviceId`，客户端需要同时保存一个本地生成的 `deviceSecret`。
7. `deviceSecret` 不展示给用户。
8. 服务端只用它做最小限度的写入校验。
9. 这不是账号体系，只是设备级参与者凭证。

## 6. 安全原则

因为没有账号体系，安全边界必须明确。

必须做到：

1. 房间必须有随机且难猜的 `roomId`。
2. 邀请链接中不能包含敏感长期凭证。
3. 设备写入位置时，只能写自己的成员记录。
4. 设备不能修改其他设备的显示名、位置、轨迹。
5. 轨迹数据必须设置保存期限。
6. 房间必须支持过期。
7. 用户必须能离开房间。
8. 用户必须能停止共享位置。
9. 页面必须清楚显示“正在共享位置”。
10. 用户拒绝位置权限时，不能反复骚扰式请求权限。

可以接受：

1. v0 阶段不做强身份认证。
2. v0 阶段通过高随机房间 ID 和设备本地凭证降低滥用风险。
3. v0 阶段不保证企业级安全。
4. v0 阶段以朋友、情侣、家庭、小团队临时共享为主要场景。

不允许：

1. 在无提示情况下后台持续上传位置。
2. 在用户关闭共享后继续上传位置。
3. 明文记录不必要的个人信息。
4. 长期保存完整轨迹。
5. 把设备密钥写死在代码中。
6. 把 Firebase 管理员密钥放到前端。

## 7. Web 实现原则

Web 优先支持手机浏览器。

必须实现：

1. 获取系统融合定位结果。
2. 获取当前位置。
3. 记录自己的移动轨迹。
4. 在 Google Map 上显示自己。
5. 在 Google Map 上显示在线的其他人。
6. 地图外显示其他人的图标或列表。
7. 点击其他人后显示对方详情。
8. 支持查看对方轨迹。
9. 支持跳转 Google Maps 导航。
10. 支持罗盘方向指针。
11. 支持显示对方距离。

注意：

1. Web 端 IMU 和罗盘权限受浏览器限制。
2. iOS Safari 对方向传感器权限要求更严格。
3. Android Chrome 支持情况相对更好，但仍需权限和能力检测。
4. 室内定位精度不稳定。
5. Web 端不能可靠使用 UWB。
6. Web 端蓝牙能力也受浏览器和系统限制。
7. Web 端只实现通用能力。
8. 精准近距离方向查找放在 App 端实现。

---

## 8. Android App 实现原则

Android App 是第一阶段原生 App。

必须实现：

1. 与 Web 一致的 UI 风格。
2. 地图显示。
3. 实时位置共享。
4. 轨迹显示。
5. Google Maps 导航跳转。
6. 罗盘方向指针。
7. IMU 方向辅助。
8. UWB 能力检测。
9. 设备支持 UWB 时，近距离精准指向对方。
10. 设备不支持 UWB 时，使用蓝牙估算距离。
11. 后台位置权限必须谨慎设计。
12. 用户必须明确知道位置正在共享。

Android 技术建议：

1. Kotlin。
2. Jetpack Compose。
3. Google Maps SDK for Android。
4. Firebase Android SDK。
5. Fused Location Provider。
6. SensorManager。
7. UWB Jetpack Library。
8. Bluetooth LE。
9. WorkManager 只用于必要的后台任务。
10. Foreground Service 只在明确需要持续共享位置时使用。

---

## 9. iOS 后期实现原则

iOS 放在后期移植。

必须保证：

1. 视觉风格和 Web / Android 一致。
2. 交互逻辑一致。
3. 数据结构一致。
4. 房间模型一致。
5. 设备即用户模型一致。

注意：

2. iOS 后台位置权限审核更严格。
3. iOS 对蓝牙和定位权限提示更敏感。
4. 后期实现时必须单独补充 iOS 设计文档。
